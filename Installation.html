Cloud Trader Pro

Enterprise-Grade Algorithmic Trading Infrastructure Installation & Deployment Guide

System Architecture Overview

Cloud Trader Pro operates under a robust, Headless Decoupled Architecture which segregates core trading mechanisms from standard user-facing controllers:

  • Headless Backend (Server): A background engine running autonomously. It maintains persistent broker links, handles live web socket market ticks, and runs algo strategies.
  • Web Dashboard (Remote): A modern browser portal served by the backend to manage credentials and track status remotely.
  • Tkinter Desktop UI (Local): A specialized desktop console for professional scalping and detailed configuration connecting directly to the server backend.

Option 1: Docker Deployment (Recommended)

Using Docker encapsulates the application, libraries, and runtime environment. It eliminates the need to manually install Python or configure dependencies on the host system.

1. Install Docker on Ubuntu Linux:

Execute the following commands to install Docker Engine and Docker Compose:

bash 
# Update package list and install prerequisites
sudo apt update && sudo apt upgrade -y
sudo apt install -y curl bzip2 tar git

# Install Docker Engine and the Compose plugin
sudo apt install -y docker.io docker-compose-v2

# Start and enable the service to boot automatically
sudo systemctl enable --now docker

1. Install Docker Desktop on Windows:

  • Download the official installer: Docker Desktop for Windows.
  • Run the installer and make sure that the option to enable the WSL 2 backend is checked.
  • Restart your computer if prompted by the installation wizard.

3. Configure App Settings

Go to the configs/ folder, rename app_settings.json.sample to app_settings.json, and configure your parameters. On Ubuntu/Linux, you can copy the sample settings and edit it directly using nano:

bash
cd /opt/cloudtraderpro
sudo cp configs/app_settings.json.sample configs/app_settings.json
sudo nano configs/app_settings.json
json 
{
    "dashboard_password": "your_secure_web_password",
    "local_api_key": "shared_secret_for_tkinter_app",
    "active_broker": "flattrade", 
    "jwt_secret": "long_random_string_for_web_tokens",
    "session_timeout_minutes": 1440,
    "server_port": 8002,
    "license_key": "your_license_key",
    "serve_static_files": true
}

⚙️ Configuration Parameters Explained:

  • dashboard_password: The password used to log in to the web browser dashboard interface. Choose a strong, unique password.
  • local_api_key: A shared secret key used to authenticate and connect the local Tkinter desktop application securely with the remote server. Both the server's app_settings.json and the client's cf_secrets.json must share this exact key.
  • active_broker: Specifies the broker to be used by the trading system. Supported values are "flattrade" or "upstox".
  • jwt_secret: A long random secret key used by the backend to sign web tokens (JSON Web Tokens) for the dashboard session. Change this to a secure random string (e.g. 32 characters) to secure your browser session tokens.
  • session_timeout_minutes: The duration (in minutes) for which your login session remains active in the browser dashboard before requiring re-authentication. The default is set to 1440 minutes (exactly 24 hours / 1 day).
  • server_port: The port number on which the FastAPI backend web server runs and listens for incoming requests. The default is 8002.
  • license_key: The cryptographically signed license key generated for your account. This is required to unlock the trading engine and activate automated strategies.
  • serve_static_files: Configures whether the FastAPI backend serves static files (true) or delegates this entirely to Nginx (false). Set this to false in production.
Important Notice

Always customize the default dashboard password, local api key, and jwt secret. The license_key field is required to cryptographically validate your software license upon server startup.

4. Boot and Manage the Server (Docker Hub)

The provided docker-compose.yml file is preconfigured to run a multi-container stack:

  • backend-api: The Python FastAPI service (runs bibhutibbb/cloudtraderpro-backend:latest).
  • frontend-nginx: The Nginx proxy serving static UI files and forwarding dynamic endpoints (runs bibhutibbb/cloudtraderpro-frontend:latest).
  • ARM64 Support: Both images support both standard 64-bit x86 (amd64) and 64-bit ARM (arm64) architectures.
Simply run the command to start the containers:

bash
sudo docker compose up -d
powershell
docker compose up -d

5. Alternative: Loading via Distributed .tar File (Offline Setup)

If you received an offline container package (e.g., cloudtraderpro_v1.tar) instead of pulling from Docker Hub, load it into Docker first before starting:

bash

6. Stopping & Removing Containers

If you need to stop the server or clean up container resources, select your OS for the appropriate commands:

Stop the running containers (keeps data intact):

bash
sudo docker compose stop

Stop and completely remove containers (frees up ports & network):

bash
sudo docker compose down

Remove containers and delete loaded images (to free up disk space):

bash
sudo docker compose down
sudo docker rmi bibhutibbb/cloudtraderpro-backend:latest bibhutibbb/cloudtraderpro-frontend:latest

Stop the running containers (keeps data intact):

Step 5: Updating the Docker Container (When a New Version is Released)

When a new version of the Cloud Trader Pro image is uploaded to Docker Hub, you can update your setup easily without losing your configurations, databases, or logs (since they are stored outside the container in your host directories).

Navigate to your deployment directory and run the standard update sequence:

bash 
cd /opt/cloudtraderpro
sudo docker compose pull
sudo docker compose up -d

Open PowerShell, navigate to your deployment directory, and pull/start the new container:

powershell 
cd C:\CloudTraderPro
docker compose pull
docker compose up -d
Linux Command Chaining Tip

If you choose to chain commands together with && on Linux, remember that sudo only applies to the command immediately following it. You must prepend sudo to both parts of the command line, for example:
sudo docker compose down && sudo docker compose up -d
Omitting the second sudo will trigger a Permission Denied socket error.

How it works & Custom Port Persistence:
  • Local Configurations Safe: Your API keys, passwords, and other credentials inside configs/app_settings.json are stored on the host machine and are never modified or overwritten by image updates.
  • Port Settings Remain Intact: If you mapped a custom host port (e.g. 8003:8002) in your local docker-compose.yml, that file will not be changed. Your custom port mapping will continue to function on the new container version.
  • Caution with Installer Script: If you update by re-running the full installer script (install.sh/install.ps1), it will download a fresh copy of docker-compose.yml from GitHub, which will overwrite your custom port settings. If you do this, you will need to re-apply your port mapping changes in docker-compose.yml.

Option 2: Local Python setup

If you wish to bypass Docker containerization and run Python directly, follow these steps.

Source Code Distribution Notice

The core application source code is proprietary and is not hosted on the public GitHub build repository. To deploy locally, you must first obtain the official application distribution package (ZIP file) by contacting the developer directly.

1. Obtain & Extract the Application Package:

  • Contact the developer to request the official Cloud Trader Pro distribution ZIP file.
  • Extract the received ZIP archive into a dedicated directory of your choice (e.g. C:\CloudTraderPro or /opt/cloudtraderpro).
  • Open your terminal or PowerShell inside the extracted directory to run the following setup commands.

Prerequisites:

  • Install Python 3.12+
  • Install the uv package manager (recommended for fast dependencies synchronization).

2. Install `uv` Package Manager:

bash
curl -LsSf https://astral.sh/uv/install.sh | sh
source $HOME/.local/bin/env
powershell
powershell -c "irm https://astral.sh/uv/install.ps1 | iex"

3. Install Tkinter & Dependencies:

On Linux systems, python's built-in Tkinter package must be installed globally:

bash
sudo apt update

4. Sync Packages & Boot Server:

Navigate to the project root and run:

bash
# Synchronize environment excluding development utilities
uv sync --no-dev

# Run backend server
uv run python run_server.py

Pre-loading Historical Backtesting Data

If you have purchased or generated historical backtesting data (saved as .parquet files), you can easily pre-load them into the system. This allows immediate backtesting without having to fetch the files over the live broker API.

Target Directories:

The files must be placed in specific directories depending on your installation type:

Directory Path
/opt/cloudtraderpro/datafetcher/historicaldatas/
Directory Path
C:\CloudTraderPro\datafetcher\historicaldatas\

How to Upload Files to a Linux VPS:

Choose one of the following methods to transfer the files from your local computer to your remote server:

SFTP Transfer (Recommended - Graphical)

  1. Download and install a free SFTP client such as FileZilla or WinSCP.
  2. Connect to your VPS using your SSH credentials (IP Address, username ubuntu, and your private key file).
  3. On the remote server side, navigate to the target directory: /opt/cloudtraderpro/datafetcher/historicaldatas/.
  4. Drag and drop your .parquet files from your local file explorer into the remote directory.

SCP Command Line Transfer

From your local machine's terminal or PowerShell, run the following command to copy the files directly:

bash
scp -i /path/to/key.pem -r /local/path/to/data/*.parquet ubuntu@YOUR_VPS_IP:/opt/cloudtraderpro/datafetcher/historicaldatas/
Critical Guidelines
  • Do Not Rename Files: The backtest engine expects files to be named exactly according to their symbol/token names (e.g., NSE_26000.parquet). Changing the names will make them invisible to the backtest parser.
  • No Server Restart Required: Because the directories are volume-bound in Docker, files are detected instantly by the backtester as soon as they are placed in the folder.

Configuring Notification Webhooks

Cloud Trader Pro has an integrated asynchronous alert engine. When trades are placed, positions are squared off, or safety guards trigger, alerts are sent instantly to your personal device. Because they run in the background, they add zero latency to your execution speed.

Step-by-Step Telegram Setup

Telegram routes messages securely using your unique Chat ID and a Bot Token. Setting it up takes less than 2 minutes and is completely private (no phone numbers are shared with the API).

  1. Create the Telegram Bot (The Sender):
    • Search for @BotFather (the official bot builder) in Telegram and press Start.
    • Send the command /newbot.
    • Follow the prompts to specify a display name (e.g., My Alerts Bot) and a unique username ending in "bot" (e.g., MyCloudTraderAlertBot).
    • Copy the **HTTP API Token** returned by BotFather (e.g., 738194:ABCde...).
  2. Activate your Bot (Anti-Spam Requirement):
    • Click your new bot's link (e.g., t.me/MyCloudTraderAlertBot) or search for its username.
    • Click Start or send a message. Note: If you skip this, Telegram's privacy shield will block the bot from sending you alerts.
  3. Retrieve your personal Chat ID (The Destination):
    • Search for @userinfobot in Telegram and press Start.
    • Copy the numeric Id returned in the chat response (e.g., 987654321).
  4. Link & Save Settings:
    • Open your Cloud Trader Pro Web Dashboard and navigate to the Application Configuration page.
    • Toggle on Telegram Bot Notifications.
    • Paste your Telegram Bot Token and Telegram Chat ID in the input fields.
    • Click Save Settings.

Step-by-Step Discord Webhook Setup

Discord Webhooks allow Cloud Trader Pro to post rich-format embedded messages straight to a designated text channel inside your own Discord Server.

  1. Generate a Discord Incoming Webhook:
    • Open Discord and navigate to your server.
    • Right-click the channel where you want notifications and select Edit Channel.
    • Go to the Integrations tab on the left and select Webhooks.
    • Click Create Webhook (or New Webhook).
    • Give the webhook a name (e.g., Trader Alerts) and click Copy Webhook URL.
  2. Save coordinates in Web App:
    • Open the Application Configuration page on the web dashboard.
    • Toggle on Discord Webhook Notifications.
    • Paste your copied Webhook URL in the corresponding field.
    • Click Save Settings.

Troubleshooting & Diagnostics

"Illegal Instruction" / NumPy Compatibility Crash

Recent compiled packages for NumPy and Pandas require advanced CPU capabilities (like AVX instruction sets). On older CPUs, this triggers compatibility crashes.

Solution: Since end-users cannot rebuild the prepackaged Docker image themselves, you must run the server using Local Python Deployment (Option 2), and force older, compatible package versions using:

bash
uv add "numpy<2.0" "pandas<2.3"

SSL Certificate Validation Errors (Windows)

If the Windows host machine fails to fetch symbols or connect to broker APIs due to SSL/TLS handshake exceptions, the local Certificate Authority store may be outdated.

Solution: Run PowerShell with Administrative Privileges and run these commands to update root certificates:

powershell
certutil -generateSSTFromWU roots.sst
certutil -addstore -f root roots.sst
del roots.sst

# Jump into container CLI shell for directory analysis

Networking & Cloudflare Tunnel

Exposing the local FastAPI endpoint to the public internet securely (with SSL/TLS) is essential for receiving broker callbacks. A Cloudflare Zero Trust Tunnel is the recommended way to set this up.

1. Obtain Tunnel Token:

  • Access your Cloudflare Zero Trust Console.
  • Go to Access -> Tunnels -> Create a Tunnel.
  • Under the "Docker" configuration guide, copy the full docker run... command (or copy just the raw token string).

2. Run Automatic Setup helper:

Open cloudflare_tunnel_command.txt in a text editor and paste the copied token or the entire command. On Ubuntu Linux, you can open and edit the file directly using nano:

bash
cd /opt/cloudtraderpro
sudo nano cloudflare_tunnel_command.txt

Save the file and run the installer script:

bash
chmod +x setup.sh
sudo ./setup.sh
powershell
# Double-click setup.bat or run:
.\setup.ps1

The helper automatically updates the .env file, establishes a docker-compose.override.yml containing the Cloudflare Tunnel container sidecar, and initiates the service stack.

3. Configure Route Hostname:

  • In the Cloudflare dashboard, edit your active tunnel and go to the Public Hostname tab.
  • Click Add a public hostname and set:
    • Subdomain: e.g., trader
    • Domain: Your registered domain (e.g., domain.com)
    • Type: HTTP
    • URL: frontend-nginx:80
  • Save settings. Now, your server dashboard is publicly available over secure HTTPS at https://trader.domain.com.

4. Broker-Side Handshake:

  • Log into the Developer Portal for your Active Broker (Flattrade or Upstox).
  • Select your App and locate the Redirect URL field.
  • Update the redirect URL according to your setup:
    • For Remote Server / Cloudflare Tunnel: https://trader.yourdomain.com/api/auth/callback
    • For Local Windows PC Setup (No Domain): http://localhost:8002/api/auth/callback (Note: If you customized your server port, replace 8002 with your custom port).
  • Critical Requirement

    Ensure there are no trailing slashes or spaces. The URL must match the broker registration exactly.

5. Multi-Subdomain API Access with Service Auth (Optional):

If you want to protect your dashboard web portal with Cloudflare Access (forcing identity provider or email logins for browser users) while still allowing external automated trading scripts or API clients to access the backend directly, you can set up Service Token Auth.

Option A: Two Subdomains (UI & API Isolation)

This is the cleanest approach. It exposes the browser UI on one subdomain and the automation API on another:

  1. Cloudflare Tunnel Setup: In your Cloudflare Tunnel dashboard, add two public hostnames pointing to the same local Nginx container port (e.g., 8002 on the host, which is mapped to port 80 inside the container):
    • ctp.shoonyatrader.in → http://localhost:8002
    • ctp-api.shoonyatrader.in → http://localhost:8002
  2. Nginx Handling: You do not need to modify Nginx configuration because Nginx is configured to listen to all incoming domains (server_name localhost; in nginx.conf) and proxy any unmatched paths to the FastAPI backend dynamically.
  3. Cloudflare Access Policies: Create two different Access Applications:
    • Dashboard App (ctp.shoonyatrader.in): Set the policy action to Allow and assign email OTP, Google Auth, or SSO rules.
    • API App (ctp-api.shoonyatrader.in): Set the policy action to Service Auth and bind it to your generated Service Token.
  4. Usage:
    • Browser Users: Navigate to https://ctp.shoonyatrader.in and complete the login. Your browser session cookies will automatically authorize all relative backend calls.
    • External Scripts: Target your automated code (e.g., Python requests) at https://ctp-api.shoonyatrader.in/api/... and pass the CF-Access-Client-Id and CF-Access-Client-Secret headers. CORS issues are completely bypassed since external command-line clients do not enforce CORS.

Option B: Single Subdomain with Path-Based Policies

If you prefer using a single subdomain:

  1. Create one Cloudflare Access Application for ctp.shoonyatrader.in.
  2. Add a Browser Allow policy (OTP/Google) covering the entire domain.
  3. Add a Service Auth Bypass policy restricted to specific path rules: api/* and ws/*. Ensure your external scripts include the CF-Access-Client-Id and CF-Access-Client-Secret headers when calling those paths.

6. Port Mapping & Nginx Internal Routing Architecture:

If you change the external port mapping in your docker-compose.yml (for example, setting 8003:80 on the host to avoid local port allocation conflicts):

  • No internal config changes are needed: Port 8003 is mapped at the Docker host level. The Nginx proxy container still listens on port 80 internally, and Uvicorn still listens on 8002 inside the backend container. The hardcoded backend-api:8002 references inside nginx.conf do not need to change because container communication happens entirely on the internal Docker network.
  • No local configuration files needed: You do not need a copy of nginx.conf on your AWS server host machine. It is copied inside the Docker image during the build process (Dockerfile.nginx) and runs entirely within the frontend container.
  • FastAPI Optimization: When running behind Nginx, it is highly recommended to edit configs/app_settings.json and set "serve_static_files": false. This tells the FastAPI server to stop serving static assets, delegating static delivery entirely to Nginx.

Tkinter Desktop Client Setup

The Tkinter desktop application (main.py) is designed to run on a local machine (Windows PC or local Linux Desktop). It cannot display a graphical interface if run directly on a headless Ubuntu Linux Server without a desktop environment; however, it can connect remotely to a backend hosted on an Ubuntu Server.

Development Status

A compiled standalone Windows .exe package is planned for future release. Currently, the Tkinter app is executed using Python. Both the Python execution method and the future standalone .exe require proper connection secrets to communicate with the server backend.

Starting the Desktop Client:

Navigate to the project root directory on your local machine and run:

bash
uv run python main.py

Configuring connection secrets (`configs/cf_secrets.json`):

Create or edit the configs/cf_secrets.json file in your local application directory to configure the remote server coordinates and bypass Cloudflare Access policies:

json
{
    "base_url": "https://trader.yourdomain.com",
    "ws_url": "wss://trader.yourdomain.com/ws",
    "use_cloudflare": true,
    "local_api_key": "must_match_app_settings_on_server",
    "CF-Access-Client-Id": "your-cloudflare-client-id.access",
    "CF-Access-Client-Secret": "your-cloudflare-client-secret-hex-string"
}
  • base_url & ws_url: The connection endpoints pointing to your server:
    • Local Python setup / SSH Port Forwarding: Use "base_url": "http://localhost:8002" and "ws_url": "ws://localhost:8002/ws" (with "use_cloudflare": false).
    • Cloudflare Tunnel Remote setup: Use "base_url": "https://trader.yourdomain.com" and "ws_url": "wss://trader.yourdomain.com/ws" (with "use_cloudflare": true).
  • local_api_key: Must match the local_api_key value in the remote server's app_settings.json.
  • CF-Access-Client-Id & CF-Access-Client-Secret: If you are protecting your domain via a Cloudflare Access Policy (SSO), you must configure a **Service Token** in your Cloudflare Zero Trust Console and paste the resulting credentials here. This allows the Tkinter client to authenticate cryptographically and bypass the browser login gate.
  • use_cloudflare: Set to true to pass the Access headers, or false if connecting locally via localhost (e.g. over SSH Port Forwarding).

Alternative Access Methods (Without a Domain)

If you do not own a custom domain, you can deploy the app using one of these three options:

Option A: Cloudflare Quick Tunnels (Random Temporary Domain)

Run a temporary public tunnel without creating an account. Add this service block to your docker-compose.yml file:

yaml
quick-tunnel:
  image: cloudflare/cloudflared:latest
  command: tunnel --url http://cloud-trader-pro:8002

Boot up the container and check the container log to locate the generated URL link:

bash
docker compose logs quick-tunnel

Look for a line containing: https://some-random-words.trycloudflare.com.

Option B: SSH Port Forwarding (Recommended for Private Access)

Keep your firewall ports locked and secure. Establish an encrypted tunnel from your local PC to the remote server by running:

bash
ssh -N -L 8002:localhost:8002 user@YOUR_SERVER_IP

Now, open http://localhost:8002 in your local browser to access the server safely.

Option C: Request a Subdomain from the Developer/Admin (Easiest)

If you do not own a domain, you can request the developer/admin to provision a custom subdomain (e.g. yourname.shoonyatrader.in) on their own network infrastructure. They can link this subdomain directly to your Cloudflare Tunnel connector.
